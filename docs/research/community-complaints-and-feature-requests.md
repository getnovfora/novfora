# Community Complaints & Feature Requests — Evidence Table

> **Project:** Hearth (working codename). **Stage A deliverable** for Section 5.4.
> **Date:** 2026-06-01. **Method:** hybrid research — live web search across operator forums (phpBB/MyBB
> communities), Hacker News, review sites (Trustpilot, Capterra), vendor docs, and technical write-ups.
> Every row carries a real source link. Claims are split into **confirmed patterns** (multiple independent
> sources) vs **weaker/one-off signals**.
> **How to read the last column:** "Our design response" is Hearth's synthesized answer; it links to the
> ADR/architecture doc where the decision is recorded and the [roadmap](../product/roadmap.md) phase that
> delivers it. These responses are the evidentiary backbone of our MVP and prioritization.

## 4. Confidence summary

- **Strongest evidence: spam as the dominant operator burden.** Corroborated by operator quotes on the
  platforms' own forums (80–100 registrations/day despite full tooling), multiple independent HN testimonies
  from people who *shut down* forums over it, Akismet's framing of forum spam as community-killing, and
  CleanTalk blocking ~195M requests in a single month. **Citable with high confidence across all
  open-source platforms** — and the reason anti-spam gets its own first-class ADR.
- **Solid evidence (4–6 sources each): mobile UX, upgrade brittleness, theming-needs-core-edits, migration
  data loss, install/ops burden, and the Invision pricing backlash.** The upgrade problem is well-attested
  enough that a *paid-services market* exists around it. SEO loss is unusually well-quantified: **−95% search
  traffic despite 700+ redirects**, and **only ~30% of URLs indexed** after a migration — both tracing to
  the C5 source rows.
- **Well-supported technical claim: MySQL full-text weakness** — a documented 60× degradation benchmark, a
  5-minute query abort, independent Sphinx benchmarks, and XenForo's own decision to sell an Elasticsearch
  add-on all converge.
- **Thinnest evidence: real-time and monetization.** Both recur as feature requests but lack strong
  operator-impact testimony or quantified cost; we treat them as Should/Could, not Must. Real-time is largely
  an *absence* (forums simply lack it) rather than an active complaint.
