# Phase 1.5 — Validation & Hardening (pre-beta) — Claude Code kickoff prompt

> Owner decision (2026-06-03): Phase 1 / Core MVP is complete; **validate the core promise before building
> more features.** This is a **fresh, adversarial** review-and-harden pass toward a private beta — NOT new
> features. Run it in a **new Code session** (don't carry build-session context — review the code as untrusted).
> **Code-vs-human split:** Code does Part 1 (security review) + Part 2 (real-host-readiness hardening + it
> *produces* the real-host runbook). The actual install on ≥2 real shared hosts is a **human step** that
> follows, using the runbook Code produces — the container has no real-host access.
> Specs to review against: [security-and-permissions.md](../../architecture/security-and-permissions.md) §1 (permissions)
> + §2 (anti-spam) + §4 (OWASP baseline); PROJECT-BRIEF §6 (operability) + §9 (security/privacy); the M5 installer
> (`app/Install/`), ADR-0020.

```
Phase 1.5 — Validation & Hardening (pre-beta). Phase 1 is complete; the owner chose to VALIDATE the core
promise (safe self-hosting on cheap hosting) before new features. This is an ADVERSARIAL review + hardening
pass, not feature work. Approach the existing code as a hostile reviewer would — assume it's untrusted.

ORIENT (read-only first): read PROJECT-STATE.md (Phase 1 complete), CLAUDE.md, PROJECT-BRIEF.md §6 + §9, and
security-and-permissions.md §1/§2/§4. Confirm the M5 baseline is green in Docker before changing anything.

MODEL/EFFORT: Opus 4.8 at xhigh throughout — every surface here (installer, permission engine, anti-spam) is
a CLAUDE.md deep-reasoning area. Work from a threat model, not a feature list.

RULES OF ENGAGEMENT: fix CLEAR-CUT issues with a regression test each. For anything that is a design change, a
real vulnerability with tradeoffs, or would alter approved behavior — STOP and report it for owner decision,
don't silently change it. Produce findings even where you don't fix. Small conventional DCO commits.

PART 1 — ADVERSARIAL SECURITY REVIEW (produce docs/SECURITY-REVIEW.md: severity-ranked findings → fix/flag):

  • Installer (highest risk): try to defeat the post-install lock — TOCTOU/races on the file marker, partial-
    install states, reaching the wizard via a crafted route after install, behavior if storage/ is unwritable.
    EnvWriter: .env injection, secret leakage in errors/logs/responses. Input validation on DB host/creds
    (SSRF/SQLi via the DB host field) and admin creds. The pre-install boot hook: does it weaken anything?
  • Permission engine: hunt privilege-escalation — can a NEVER be bypassed? group-merge / primary-vs-secondary
    edge cases; cache poisoning or stale-after-change windows beyond the ban-version fix; guest/banned handling;
    does the inspector leak anything it shouldn't?
  • Anti-spam: bypass paths — honeypot/timing evasion, CAPTCHA-degradation abuse, StopForumSpam-degradation
    abuse, trust-level promotion gaming, rate-limit bypass; can a gated (TL0) author get links/images rendered?
    registration_checks PII handling + retention.
  • Cross-cutting OWASP (security §4): IDOR on every gated action (act on another user's post/topic/attachment/
    notification by ID?); CSRF on all state-changing routes; XSS through the canonical→sanitize boundary across
    the FULL node set + signatures + custom profile fields + mentions; SQLi (any raw queries?); upload handling
    (type/size/path-traversal, attachment-stream authz); mass-assignment audit on ALL models (M5 already found
    a Fillable bug — audit the rest); CSP/session/cookie hardening; audit-log completeness on security events.
  • Supply chain: composer audit + npm; confirm pinned versions + the MIT/Apache-only constraint (ADR-0015).

PART 2 — REAL-HOST READINESS + THE RUNBOOK (so the human real-host test goes smoothly):

  • A `novfora:doctor` preflight (extend RequirementChecker) that detects shared-host gotchas: PHP version +
    required/optional extensions, DISABLED functions (does any baseline path rely on proc_open/exec/symlink?),
    open_basedir, writable paths, session/cache/queue drivers, outbound mail capability, whether symlink
    (storage:link) actually works — and a COPY-BASED fallback for public storage when symlink is disabled, plus
    coarse-cron correctness. Harden the baseline path to need none of the commonly-disabled functions.
  • Produce docs/REAL-HOST-VALIDATION.md — a concrete runbook a human follows to install NovFora on ≥2 real
    shared hosts (e.g., a cPanel PHP 8.3 host + a budget host): exact steps, the one cron line, mail + storage
    setup, what to verify at each step, and how to capture/report any failure. Include the host-compatibility
    checklist.

DEFINITION OF DONE: docs/SECURITY-REVIEW.md (findings + what was fixed vs flagged); clear-cut fixes applied
WITH tests; `novfora:doctor` green on the Docker baseline + documented fallbacks (symlink/mail); docs/REAL-HOST-
VALIDATION.md shipped. The full M0–M5 suite STAYS green; Pint/Larastan/composer-audit clean. Report back: the
findings summary + the runbook, and explicitly list anything flagged for owner decision.

SCOPE FENCE: no new product features (nothing from Phase 2/3/4). This is review + hardening + tooling +
docs only. Keep the nullable tenant_id seam untouched.
```

## After this session — the human step, then the fold-back

1. **You (or whoever ops it)** run `docs/REAL-HOST-VALIDATION.md` on **≥2 real shared hosts** — provision them,
   run the no-SSH installer end-to-end, and report what breaks (with the `novfora:doctor` output).
2. **Cowork** triages the real-host findings + the SECURITY-REVIEW flags, and preps a focused fix kickoff —
   after which NovFora is ready for a **private beta**. (A truly independent third-party security review and the
   full WCAG/load/i18n pass remain worthwhile before a *public* 1.0 — that's the Path C work, for later.)
